# Датасет камер: `cameras_raw.csv`

Файл `cameras_raw.csv` содержит ~40 000 строк с данными о камерах. Это **исходный, заведомо «грязный» датасет**, который должен быть очищен и загружен в БД бэкенда.

## Формат

CSV с разделителем `,` и первой строкой-заголовком. Пример (не реальная строка файла):

```csv
id,server_id,name,ip_address,rtsp_url,status,resolution,fps,container_id,is_active,created_at,updated_at,last_heartbeat_at,location,tags
1,12,Entrance Cam 1,10.0.5.10,rtsp://10.0.5.10/stream/R1,RUNNING,1920x1080,25,ab12cd34ef56,true,2023-01-10T12:30:00Z,2023-06-01T10:20:00Z,2023-06-01T10:19:30Z,DC-1 / Floor 1 / Rack 3,"entrance;critical"
```

Колонки:

1. `id` — числовой идентификатор камеры (внутренний).
2. `server_id` — числовой идентификатор сервера (связь с сущностью Server).
3. `name` — имя камеры.
4. `ip_address` — IP-адрес камеры.
5. `rtsp_url` — RTSP-URL потока.
6. `status` — строка, ожидаемо одно из: `RUNNING`, `STOPPED`, `ERROR` (но в файле есть и некорректные значения).
7. `resolution` — строка, например `1920x1080`.
8. `fps` — кадров в секунду (ожидается целое число).
9. `container_id` — идентификатор Docker-контейнера.
10. `is_active` — логическое значение (`true`/`false`), но в файле встречаются и другие варианты.
11. `created_at` — дата/время создания.
12. `updated_at` — дата/время обновления.
13. `last_heartbeat_at` — последняя отметка «живости» (может быть пустой).
14. `location` — строка с расположением.
15. `tags` — строка с тэгами, разделёнными `;` (может быть пустой).

## Типичные проблемы в данных

Датасет специально сгенерирован с ошибками. Среди них, в частности:

- дубликаты по `id` и по комбинации полей (`server_id`, `name`);
- некорректные IP-адреса (`999.999.0.1`, `not_an_ip` и т.п.);
- невалидные значения `status` (`UNKNOWN`, пусто и т.п.);
- некорректные даты (`2023-99-99`, нестандартные форматы, пустые значения);
- строки с `fps`, где вместо числа текст (`N/A`, `-1` и т.п.);
- лишние пробелы в полях, особенно в `name`, `location`, `tags`;
- разные варианты записи булевых значений (`TRUE`, `False`, `0`, `1`, пусто).

Задача бэкенда — реализовать надёжный процесс:

1. Чтения файла кусками/батчами.
2. Валидации и нормализации данных.
3. Загрузки корректных записей в БД.
4. Логирования статистики и ошибок.

Конкретная стратегия обработки дубликатов и невалидных записей остаётся на усмотрение исполнителя, но должна быть описана в коде или документации.
