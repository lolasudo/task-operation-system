# Задание для фронтенда: Панель управления серверами и камерами

## Цель

Реализовать одностраничное веб-приложение (SPA) для администраторов, которое работает поверх API сервиса Docker Camera Management и позволяет:

- просматривать список серверов и камер;
- выполнять CRUD-операции для серверов и камер;
- запускать действия (старт/стоп/рестарт);
- загружать и просматривать результат импорта CSV (через бэкенд);
- удобно работать как на десктопе, так и на мобильном устройстве.

Требования ориентированы на индустриальные практики, но без чрезмерной формализации.

---

## Технологические требования

1. **Фреймворк**
   - Любой современный SPA-фреймворк: React, Vue, Angular, Svelte и т.п.
   - Сборка через стандартный инструмент (Vite, CRA, Angular CLI и т.д.).

2. **Структура проекта**
   - Разделение на модули/фичи (страницы, компоненты, сервисы для работы с API).
   - Выделенный слой для общения с API (например, `api/` или `services/`), без прямых `fetch`/`axios`-вызовов из компонентов.
   - Разумное использование состояния (локальное состояние компонентов + при необходимости глобальное состояние — Zustand, Redux, Pinia и т.п.).

3. **Работа с API**
   - Базироваться на спецификации `backend/openapi.yaml`.
   - Использовать переменную среды/конфигурации для `BASE_URL` API.
   - Обрабатывать состояния `loading`, `error`, `empty state`.
   - Отображать сообщения об ошибках пользователю в понятном виде.

4. **UI и UX**
   - Адаптивный интерфейс (desktop + mobile):
     - Desktop: ширина ≥ 1280px.
     - Mobile: ширина ≈ 360–414px.
   - Использование Flexbox/Grid для основных лэйаутов.
   - Можно использовать UI-библиотеку (MUI, AntD, PrimeVue, Bootstrap и т.п.) или реализовать компоненты самостоятельно.
   - Единый базовый визуальный стиль: шрифты, отступы, заголовки, кнопки.

5. **Кросс-браузерность**
   - Поддержка актуальных версий браузеров: Chrome, Firefox, Edge, Safari (последние 2 мажорные версии).
   - На мобильных — актуальные версии Chrome (Android) и Safari (iOS).

6. **Код и качество**
   - Понятная структура файлов и папок.
   - Осмысленные имена компонентов и переменных.
   - Минимальное, но достаточное количество комментариев (почему, а не что).
   - Линтер/форматтер (ESLint + Prettier или аналог).

## Основные страницы и сценарии

### 1. Авторизация

- Простая форма логина:
  - поля: `username`, `password`;
  - после успешного логина — сохранение токена (например, в `localStorage` или в памяти) и редирект в основное приложение.
- Защищённые маршруты:
  - все страницы кроме логина требуют наличия валидного токена;
  - при отсутствии токена — редирект на страницу логина.

### 2. Список серверов

- Таблица/список со следующими полями (минимум):
  - имя сервера;
  - hostname;
  - IP;
  - статус;
  - расположение;
  - количество камер (можно получать отдельным запросом или включать в ответ API).
- Фильтры и поиск:
  - по статусу;
  - по локации;
  - текстовый поиск по имени/hostname.
- Пагинация:
  - переключение страниц;
  - выбор размера страницы (например, 10/25/50).

Действия из списка:

- открыть детали сервера;
- перейти к списку камер на сервере;
- кнопка «Перезапустить» сервер (вызов `POST /servers/{id}/actions/restart`).

### 3. Карточка сервера

- Отдельная страница с подробной информацией о сервере.
- Возможность редактировать данные сервера (форма) и сохранять изменения (PUT/PATCH).
- Отображение последних операций/ошибок (если бэкенд предоставляет такую информацию).

### 4. Список камер

- Таблица/список камер со столбцами:
  - имя;
  - сервер;
  - IP;
  - статус;
  - разрешение;
  - fps;
  - `is_active`;
  - дата/время последнего heartbeat.
- Фильтры:
  - по статусу;
  - по серверу;
  - по активности (`is_active`);
  - текстовый поиск (имя, IP, RTSP-URL).
- Пагинация аналогично списку серверов.

Действия:

- открыть карточку камеры;
- запустить/остановить/перезапустить камеру (кнопки «Start», «Stop», «Restart»).

### 5. Карточка камеры

- Детальная информация (все поля сущности).
- Возможность редактирования полей (кроме `id`, `created_at` и т.п.) и сохранения.
- Возможность смены сервера, к которому привязана камера (селект со списком серверов).
- Отображение истории/лога операций по камере (при наличии такой информации от бэкенда или в упрощённом виде — хотя бы последние статусы).

### 6. Импорт CSV (через бэкенд)

- Страница/секция «Импорт данных»:
  - кнопка «Загрузить CSV» (с выбором файла);
  - загрузка файла на бэкенд (например, `POST /import/cameras` или аналогичный endpoint, который вы реализуете со стороны бэкенда);
  - отображение статуса импорта:
    - всего строк;
    - успешных записей;
    - ошибок (можно список с paginaton или усечённую таблицу);
    - краткое резюме по типам ошибок.
- Пользовательский фидбек:
  - состояние «идёт импорт»;
  - успешное завершение;
  - отображение ошибок, если импорт не удался.

## UX-детали

- Визуальное разделение действий, которые могут привести к потере данных (удаление сервера/камеры) — модальные окна с подтверждением.
- Отображение спиннеров/скелетонов при загрузке больших таблиц.
- Сообщения об ошибках и успехе (toast/notification).

---

## Адаптивность

Минимальные требования:

- На десктопе таблицы занимают доступную ширину экрана, есть горизонтальные отступы.
- На мобильных:
  - вертикальный скролл;
  - таблицы могут быть преобразованы в карточки (одна камера/сервер = одна карточка с основными полями);
  - крупные зоны для клика (кнопки, ссылки);
  - меню/навигация должны быть удобны (бургер-меню, нижняя навигация и т.п.).
