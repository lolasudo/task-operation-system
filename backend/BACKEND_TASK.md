# Задание для бэкенда: Docker Camera Management API

## Цель

Реализовать промышленный REST API для управления серверами и камерами, запущенными в Docker, с поддержкой:

- CRUD-операций для сущностей **Server** и **Camera**;
- служебных действий (старт/стоп/рестарт) над камерами и серверами;
- загрузки и очистки большого CSV-файла с данными о камерах;
- аутентификации и авторизации;
- логирования, мониторинга и тестирования.

API должно соответствовать спецификации `openapi.yaml` (расширение допускается, но не нарушение).

---

## Технологические требования

Язык и стек можно выбрать самостоятельно (Java/Spring Boot, .NET, Node.js/NestJS, Python/FastAPI, Go и т.д.), но решение должно соответствовать типичным индустриальным практикам:

1. **Архитектура**
   - Чёткое разделение слоёв: контроллеры / слой приложения / доменный слой / инфраструктура (репозитории, клиенты).
   - Отсутствие «утечки» инфраструктурных деталей в доменную модель.
   - Явное DTO между API и доменной моделью (запросы/ответы).

2. **Хранение данных**
   - Реляционная БД (рекомендуется PostgreSQL).
   - Обязательны миграции (Flyway/Liquibase или аналог).
   - Первичные и внешние ключи, индексы на ключевые поля (поиск, фильтрация).

3. **Конфигурация и окружения**
   - Конфигурация через переменные окружения / конфигурационные файлы.
   - Разделение окружений: `local`, `test`, `prod` (достаточно логического разделения).
   - Возможность запуска через `docker-compose` (минимум: backend + db).

4. **Логирование и мониторинг**
   - Структурированные логи (уровни: INFO, WARN, ERROR).
   - Логирование входящих запросов (минимум метод, путь, статус, время выполнения).
   - Логирование ключевых бизнес-событий (создание/изменение/удаление камер и серверов).
   - Health-check endpoint (`GET /health`) для проверки состояния сервиса.
   - (Опционально) Метрики (количество запросов, время ответа и т.п.).

5. **Безопасность**
   - JWT-аутентификация (Bearer token) для всех эндпоинтов, кроме `/health`.
   - Минимум две роли: `ADMIN` и `VIEWER`.
     - `VIEWER` может только читать данные (GET).
     - `ADMIN` может выполнять любые операции (CRUD + действия).
   - Защита от типичных проблем: валидация входных данных, ограничение размера запросов, корректная обработка ошибок.

6. **Тестирование**
   - Юнит-тесты для бизнес-логики (минимум happy-path + 1–2 негативных сценария на каждый ключевой use-case).
   - Интеграционные тесты API (можно с in-memory БД или test-контейнерами).
   - Покрытие тестами критичных частей (загрузка CSV, действия над камерами/серверами).

## Доменная модель

### Server

Сервер, на котором запущен Docker и контейнеры с камерами.

Обязательные поля (можно расширять):

- `id` (UUID или bigint, PK)
- `name` (строка, уникальная в рамках системы)
- `hostname` (строка, DNS-имя или FQDN)
- `ip_address` (строка, IPv4/IPv6)
- `location` (строка, например, «МЦОД-1 / Ряд 3 / Стойка 12»)
- `docker_version` (строка)
- `status` (enum: `ONLINE`, `OFFLINE`, `MAINTENANCE`)
- `created_at` (timestamp)
- `updated_at` (timestamp)
- `labels` (опционально, JSON/тэги)

### Camera

Камера, которая в реальности является контейнером, поднятым на одном из серверов.

Обязательные поля (можно расширять):

- `id` (UUID или bigint, PK)
- `server_id` (FK на `Server`)
- `name` (строка)
- `ip_address` (строка, IPv4/IPv6)
- `rtsp_url` (строка)
- `status` (enum: `RUNNING`, `STOPPED`, `ERROR`)
- `resolution` (строка, например `1920x1080`)
- `fps` (целое число)
- `container_id` (строка, идентификатор Docker-контейнера)
- `is_active` (boolean)
- `created_at` (timestamp)
- `updated_at` (timestamp)
- `last_heartbeat_at` (timestamp, может быть null)
- `location` (строка)
- `tags` (опционально, список/JSON)

## Функциональные требования

### 1. CRUD для Server

Реализовать эндпоинты (см. детали в `openapi.yaml`):

- `GET /api/v1/servers` — список серверов с пагинацией и фильтрацией:
  - фильтрация по `status`, `location`, подстроке в `name`/`hostname`;
  - пагинация через `page` и `page_size`.
- `POST /api/v1/servers` — создание сервера.
- `GET /api/v1/servers/{id}` — получение сервера по id.
- `PUT /api/v1/servers/{id}` — полное обновление.
- `PATCH /api/v1/servers/{id}` — частичное обновление (опционально, но желательно).
- `DELETE /api/v1/servers/{id}` — логическое удаление (soft-delete) или физическое, на ваш выбор (должно быть явно описано в коде/документации).
- `GET /api/v1/servers/{id}/cameras` — список камер на конкретном сервере.

### 2. CRUD для Camera

- `GET /api/v1/cameras` — список камер с пагинацией и фильтрацией:
  - по `status`, `server_id`, `is_active`, строке поиска по `name`/`ip_address`/`rtsp_url`.
- `POST /api/v1/cameras` — создание камеры.
- `GET /api/v1/cameras/{id}` — получение камеры.
- `PUT /api/v1/cameras/{id}` — полное обновление.
- `PATCH /api/v1/cameras/{id}` — частичное обновление (опционально, но желательно).
- `DELETE /api/v1/cameras/{id}` — логическое или физическое удаление.

### 3. Операции над серверами и камерами

Эндпоинты действий (мутаторы состояния):

- `POST /api/v1/servers/{id}/actions/restart` — перезапуск всех контейнеров камер на сервере (логика на ваше усмотрение).
- `POST /api/v1/cameras/{id}/actions/start`
- `POST /api/v1/cameras/{id}/actions/stop`
- `POST /api/v1/cameras/{id}/actions/restart`

Фактическая интеграция с Docker:

- Либо сделать полноценный клиент к Docker Engine API.
- Либо реализовать слой-адаптер с «заглушкой» (mock), который:
  - имитирует длительность операции (например, `sleep` 200–500 мс);
  - случайно генерирует «ошибки» по заданной вероятности;
  - меняет статус камеры/сервера.

Главное — чётко изолировать этот слой, чтобы его можно было подменить реальным Docker-клиентом.

### 4. Загрузка и очистка CSV

Исходный файл: `data/cameras_raw.csv` (≈40 000 строк). Он содержит различные ошибки и несоответствия:

- дублирующиеся камеры;
- некорректные IP-адреса;
- невалидные значения `status`;
- некорректные даты/формат времени;
- пустые обязательные поля;
- лишние пробелы, «грязные» строки и т.п.

Требуется реализовать функцию (скрипт/endpoint/фоновую задачу), которая:

1. Забирает CSV-файл с диска (путь должен быть конфигурируемым).
2. Валидирует и очищает данные (минимум):
   - отбрасывает заведомо невалидные записи;
   - нормализует формат дат;
   - обрезает лишние пробелы;
   - приводит значения к нужным enum’ам/типам;
   - обрабатывает дубликаты (стратегия на ваше усмотрение, но должна быть описана).
3. Загружает очищенные данные в БД.
4. Логирует статистику:
   - количество строк всего;
   - количество успешно загруженных камер;
   - количество отклонённых записей с указанием причин.

Рекомендуется реализовать это так, чтобы обработка больших файлов оставалась устойчивой (стриминговый парсинг, батчи и т.п.).

### 5. Формат ошибок

Все ошибки API должны возвращаться в едином формате, например:

```json
{
  "code": "VALIDATION_ERROR",
  "message": "Некорректные параметры запроса",
  "details": {
    "field": "ip_address",
    "reason": "Invalid IPv4 address"
  }
}
```

Необходимо обработать минимум:

- 400 — ошибки валидации;
- 401 — неавторизованный доступ;
- 403 — недостаточно прав;
- 404 — сущность не найдена;
- 409 — конфликт (например, дубликат `name`);
- 500 — внутренняя ошибка сервера.

## Нефункциональные требования

- Время ответа большинства запросов — до 500 мс при разумном размере данных.
- API должно быть детерминированным и предсказуемым с точки зрения контрактов.
- Код должен быть читаемым, с понятными именами и минимальными комментариями «почему», а не «что».
- Должна быть краткая инструкция по запуску (желательно с `docker-compose`) и описанием основного функционала API.
